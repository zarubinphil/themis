# Скилл: составление процессуальных документов

## Когда активируется

Любой запрос на составление судебного документа:
- исковое заявление, отзыв, возражения, ходатайство
- апелляционная или кассационная жалоба
- заявление об обеспечении иска, письменные объяснения
- "хочу подать в суд", "помоги написать иск", "как ответить на иск"

---

## ПРИНЦИП РАБОТЫ

Все тяжёлые задачи (чтение файлов, поиск практики, анализ позиции) выполняются
агентами заранее и кешируются в `01_context/`. Скилл читает кеш, а не сырые файлы.

```
Дорого: читать все файлы из 00_intake/ каждый раз
Дешево: читать один knowledge-map.md (уже готовый конспект)
```

---

## ПОРЯДОК РАБОТЫ

### ШАГ 0. НАЙДИ ДЕЛО

Прочитай `cases/_index.md`. Определи путь: `cases/{клиент}/{дело}/`.
Если дело не указано — спроси.

---

### ШАГ 0.5. ПРОВЕРЬ МАРКЕРЫ КОНСЕНСУСА (обязательно перед всем остальным)

**Шаг 0.5.А — определи уровень дела.**

Прочитай первые 60 строк `cases/{клиент}/{дело}/01_context/knowledge-map.md`.

Если файл отсутствует — СТОП:
```
❌ knowledge-map.md — карта знаний не построена.
Запусти: case-mapper → дождись маркера КАРТА ГОТОВА ✓ → вернись сюда.
```

Найди в файле блок `## ОХОТНИКИ — РЕШЕНИЕ ФЕМИДЫ`. Извлеки уровень: **L1 / L2 / L3**.

Если блок отсутствует — считай уровень **L2** (безопасный дефолт).

**Шаг 0.5.Б — проверь маркеры согласно уровню.**

| Уровень | Требуемые маркеры |
|---------|-------------------|
| L1 | `## КАРТА ГОТОВА ✓` в knowledge-map.md + `## СОВЕТ ЗАВЕРШЁН` в practice.md |
| L2/L3 | Все три: knowledge-map.md + practice.md + positions.md |

Прочитай первые 5 строк каждого нужного файла. Проверь наличие маркера.

**Если хотя бы один маркер отсутствует — СТОП.** Сообщи:

```
Документ не может быть составлен. Уровень дела: L[X].
Не завершены обязательные этапы:

❌ knowledge-map.md — карта знаний не построена
   → Запусти: case-mapper

❌ practice.md — совет по практике не проведён
   → L1: запусти охотника-тактика → тактик роя → practice.md
   → L2/L3: запусти охотников → /askacouncil → practice.md

❌ positions.md — совет юристов не завершён (только L2/L3)
   → Запусти: /position-council → positions.md
```

Показывай только недостающие маркеры.

**Если все нужные маркеры есть** — продолжай:
```
✓ knowledge-map.md — карта готова
✓ practice.md — совет завершён
✓ positions.md — согласовано (L2/L3)
Уровень: L[X]. Начинаю составление.
```

---

### ШАГ 1. ЧИТАЙ КЕШИ

Все маркеры проверены на Шаге 0.5. Читай файлы:

```
cases/{клиент}/{дело}/01_context/knowledge-map.md  ← карта материалов дела
cases/{клиент}/{дело}/01_context/practice.md        ← судебная практика
cases/{клиент}/{дело}/01_context/positions.md       ← правовая позиция (если L2/L3)
knowledge/redlines.md                               ← уроки из прежних правок доверителя
```

**ОБЯЗАТЕЛЬНО перед составлением прочитай `knowledge/redlines.md`** — это уроки из
правок, которые доверитель уже вносил в прежние документы (и по содержанию, и по
форматированию). Применяй усвоенные правила, чтобы не повторять те же огрехи.

К сырым файлам из `00_intake/` обращайся только за конкретной деталью,
которой нет в карте. Не читай все файлы подряд.

---

### ШАГ 2. НОРМАТИВНАЯ БАЗА

Прочитай `.claude/skills/doc-drafter/LEGAL_RU_SYSTEM.md`.

Определи применимый кодекс и извлеки для данного документа:
- Обязательные реквизиты (раздел 1, 2 или 3 в зависимости от типа)
- Требования к приложениям
- Размер госпошлины и расчёт (раздел 6)
- Льготы по госпошлине (раздел 6.4)
- Подсудность и подведомственность (раздел 8)
- Специальные требования для семейных дел (раздел 5) — если применимо

Зафикси извлечённые данные кратко.

---

### ШАГ 3. ПОЗИЦИЯ ДЛЯ ДОКУМЕНТА

На основе прочитанных кешей (шаг 1) и нормативной базы (шаг 2):

- Если `positions.md` есть — используй как основу, адаптируй под конкретный документ
- Если `positions.md` нет — построй позицию здесь, опираясь на knowledge-map.md

Позиция:
- Каждый аргумент вытекает из предыдущего
- Нет внутренних противоречий
- Только факты, подтверждённые knowledge-map.md
- Каждый довод оппонента получает прямой контрдовод

---

### ШАГ 4. ПРОВЕРОЧНЫЙ ЧЕКЛИСТ

Перед созданием файла — пройди и сообщи пользователю результат:

- [ ] Нет утверждений о фактах вне knowledge-map.md или чата
- [ ] Все реквизиты судебных актов взяты из practice.md (не из памяти)
- [ ] Нет внутренних противоречий между аргументами
- [ ] Позиция последовательна от начала до конца
- [ ] Каждый довод оппонента получил контрдовод
- [ ] Соблюдены обязательные реквизиты по LEGAL_RU_SYSTEM.md
- [ ] Госпошлина рассчитана верно (если применимо)
- [ ] Формулировки норм закона воспроизведены дословно (не перефразированы)
- [ ] Цитаты из судебных актов и Пленумов — без изменений

Только после прохождения всех пунктов — создавай файл.

> **Совет:** после создания черновика можно запустить агента `doc-reviewer`
> для независимой проверки перед финальной версией.

---

### ШАГ 5. СОЗДАНИЕ DOCX-ФАЙЛА

**Оформление — по эталону шаблона.docx:**

Перед созданием файла прочитай оба справочника:
- `.claude/skills/doc-drafter/DOCX_FORMATTING.md` — точные значения для каждого типа параграфа (pt, cm), python-docx рецепты, микротипография.
- `.claude/skills/doc-drafter/CONTENT_DESIGN.md` — оптимальный объём документа, иерархия заголовков, первый абзац, CRAC, определённые термины, блок-цитаты.

Краткая сводка:
- Шрифт: **Times New Roman** везде (явно в каждом run)
- Размеры: **14 pt** — заголовок документа; **13 pt** — подзаголовок и дата шапки; **12 pt** — всё остальное
- Межстрочный интервал: **1.15×** (docDefaults: `w:line="276" w:lineRule="auto"`)
- Поля: левое 30 мм, правое 15 мм, верхнее 20 мм, нижнее 20 мм
- Абзацный отступ основного текста: **1.25 см**
- Выравнивание основного текста: **по ширине (JUSTIFY)**
- Заголовок раздела (I./II./III.): sb=12pt, sa=6pt, жирный, JUSTIFY
- Подраздел (1./2./3.1.): sb=6pt, sa=3pt, жирный, JUSTIFY
- ПРОШУ: центр, жирный, sa=6pt
- ПРИЛОЖЕНИЯ: sb=12pt, sa=6pt, жирный, JUSTIFY
- Подпись: ФИО + пробелы + дата в одном параграфе, sb=6pt, sa=6pt, LEFT
- Курсив и подчёркивание — не используются
- Жирный внутри абзацев — для ссылок на нормы, сумм, дат, ключевых терминов

**Структура документа:**
```
1. Шапка: суд, номер дела, стороны, наименование документа, заявитель
2. Фактические обстоятельства
3. Правовая позиция (доводы → нормы → практика)
4. Просительная часть
5. Перечень приложений
6. Дата и подпись
```

**Язык:** /humanizer — русский, официально-деловой стиль, живой профессиональный текст.
Вместо ё — е. Текст понятен человеку без юридического образования.

**Абсолютный запрет:** нельзя перефразировать, сокращать или редактировать:
- тексты статей законов (ГПК, АПК, КАС, НК, ГК, СК и др.)
- формулировки постановлений Пленума ВС РФ и КС РФ
- цитаты из судебных актов

Эти тексты воспроизводятся **дословно** или не воспроизводятся вовсе.

**Куда сохранять — строгое разделение:**

```
03_drafts/                          ← ТОЛЬКО рабочие файлы (черновики, итерации)
  {документ}_v1.md                  ← первый черновик
  {документ}_v1.docx                ← для промежуточной проверки
  {документ}_v2.md                  ← после правок doc-reviewer

02_hearings/ДД-ММ-ГГГГ_название/   ← ТОЛЬКО финальные файлы (после ГОТОВ К ПОДАЧЕ)
  {документ}.md                     ← финальный Markdown (источник истины)
  {документ}.docx                   ← финальный Word (для /finalize)
  {документ}.pdf                    ← после /finalize (PDF с подписью)
```

Правило: в `02_hearings/` попадают **только** файлы с вердиктом Кони «ГОТОВ К ПОДАЧЕ».
Черновики и промежуточные версии остаются в `03_drafts/` навсегда.

> Если событие ещё не создано → создай через `/new-event` перед финальным сохранением.
> Если заседание неизвестно → всё в `03_drafts/`, перемести в hearings когда появится событие.

После финального сохранения — обнови `_event.md` и `_case.md`.
Для PDF с подписью: запусти `/finalize {путь_к_docx}`.

## РАСПРЕДЕЛЕНИЕ ЗАДАЧ ПО АГЕНТАМ

| Что делает | Кто | Файл-результат | Когда |
|------------|-----|----------------|-------|
| Читает материалы дела, строит конспект | `case-mapper` (рой) | `knowledge-map.md` ✓ | Шаг 1 |
| Ищет практику в нашу пользу | `practice-hunter-classic` (рой) | `hunter_classic.md` | Шаг 2 |
| Ищет практику против нас | `practice-hunter-skeptic` (рой) | `hunter_skeptic.md` | Шаг 2 (L2/L3) |
| Ищет процессуальную практику | `practice-hunter-tactical` (рой) | `hunter_tactical.md` | Шаг 2 |
| Синтезирует практику (L2/L3) | `/askacouncil` | `practice.md` ✓ | Шаг 3 |
| Вырабатывает позицию (L2/L3) | `/position-council` | `positions.md` ✓ | Шаг 4 |
| Составляет документ | doc-drafter (этот скилл) | `.md` + `.docx` | Шаг 5 |
| Проверяет черновик | `doc-reviewer` | отчёт → правки | Шаг 6 |

## ОГРАНИЧЕНИЯ

- Никогда не выдумывай судебные решения, номера дел, даты и реквизиты
- Не делай утверждений о содержании документов, не прочитав их или knowledge-map.md
- Если документ упоминается, но не приложен и не отражён в карте — запроси
- Не создавай файл без прохождения чеклиста на шаге 4
