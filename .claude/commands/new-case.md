# Команда: /new-case

Создаёт полную структуру папок для нового дела и регистрирует его в индексе.

## Использование

```
/new-case {клиент} {дело}
```

**Примеры:**
```
/new-case romashka-ooo vs-supplier-2024
/new-case ivanov-sergey razvod-2025
/new-case techstroy-ao bankrotstvo-2025
```

---

## Что делает

### 0. Маршрутизация клиента (ПЕРВЫЙ шаг)

Перед созданием папки — прочитать `cases/_clients.md`:

1. Найти клиента по ФИО / псевдониму / ИНН / названию компании
2. Если найден → использовать **его канонический folder** (не придумывать новый)
3. Если это компания с установленным владельцем (✓) → использовать папку **владельца**
4. Если компания с ⚠ (владелец неизвестен) → предупредить пользователя, использовать папку компании временно
5. Если не найден → создать новую папку, добавить клиента в `_clients.md`

**Если клиент уже существует** → прочитать `cases/{client}/_client.md`:
- Личные данные, семья, бизнес, имущество — подтянуть в `_case.md` автоматически
- Сообщить: «Клиент найден. Активных дел: N. Профиль загружен.»
- Проверить `cases/_relationships.md` — есть ли связи с оппонентом нового дела

### 1. Параллельно: структура + инбокс рабочего стола

Выполнять одновременно:

**А) Создаёт структуру папок** (ниже)

**Б) Перемещает все файлы из `~/Desktop/inbox/`** в `00_intake/` нового дела:
- `mv` всех файлов (не директорий) из `~/Desktop/inbox/` → `cases/{клиент}/{дело}/00_intake/`
- Директорию `unsorted/` внутри inbox не трогать
- Сообщить список перемещённых файлов в итоге
- Не спрашивать подтверждения — перемещать автоматически

### 1a. Создаёт структуру папок

```
cases/{клиент}/{дело}/
├── _case.md
├── 00_intake/
│   └── .gitkeep
├── 01_context/
│   ├── _working/          ← временные файлы роя
│   ├── _practice/         ← итоговые файлы охотников
│   └── _council/          ← протоколы советов
├── 02_hearings/
│   └── .gitkeep
├── 03_drafts/
│   └── .gitkeep
└── 04_archive/
    └── .gitkeep
```

**ВАЖНО:** `knowledge-map.md`, `practice.md`, `positions.md` НЕ создаются здесь.
Они создаются ТОЛЬКО workflow агентами после консенсуса (маркеры обязательны).

### 2. Создаёт `_case.md` по шаблону

```markdown
# {Клиент читаемое} | {Дело читаемое}

## Статус
- **Стадия:** Подготовка
- **Суд:** —
- **Номер дела:** —
- **Судья:** —
- **Следующее заседание:** —

## Стороны
- **Клиент (роль):** —, ИНН —
- **Представитель:** —
- **Оппонент (роль):** —, ИНН —
- **Представитель оппонента:** —

## Суть спора
_Краткое описание — что произошло, кто стороны, в чём спор_

## Применимое право
_ГПК / АПК / КАС / СК + ключевые статьи_

## Хронология событий (кратко)
_Добавляй по мере развития дела_

## Материалы дела
- `00_intake/` — исходники от клиента
- `01_context/knowledge-map.md` — карта знаний (case-mapper)
- `01_context/practice.md` — судебная практика (/askacouncil)
- `01_context/positions.md` — правовая позиция (/position-council)

## Журнал событий
| Дата | Событие | Результат |
|------|---------|-----------|
| {дата создания} | Дело открыто | — |
```

### 3. Обновляет три реестра

- `cases/_index.md` — новая строка: `| {Клиент} | {Дело} | {папка}/{дело} | — | — | — | — | Активно |`
- `cases/{клиент}/_client.md` → раздел "Активные Дела" — новая строка с делом
- `cases/_clients.md` → если новый клиент — новая строка в "Физические лица" или "Компании"

### 4. Выводит итог

```
✓ Дело создано: cases/{клиент}/{дело}/
  Следующий шаг: положи исходники в 00_intake/ → запусти case-mapper
  Заполни _case.md: стороны, суть спора, применимое право
  Workflow: case-mapper → охотники → /askacouncil → /council → doc-drafter
```
